Présentation

Le Syndicat des Eaux de Gévaudan (SEG) s’est donné pour mission le captage, le traitement et la distribution de l’eau potable pour satisfaire les usagers répartis sur le territoire des communes regroupées au sein d’un syndicat de communes.


Le service public d’eau est géré en régie : son organisation et son fonctionnement sont assurés directement par le syndicat de communes, qui conserve ainsi une maîtrise complète de sa gestion avec ses propres moyens matériels, humains et financiers. Les communes ont la responsabilité complète des investissements, du fonctionnement des services des eaux, des relations avec les usagers, comme l'émission des factures d'eau et leur recouvrement.

Contexte



Le responsable du service des eaux souhaiterait à terme avoir la possibilité d’afficher le degré d’anomalie de chaque commune afin de repérer rapidement les origines de pertes en eau depuis l’application Web.
Les agents communaux doivent disposer d’une application mobile pour les relevés des compteurs d’eau (vannes).

Mission





Mise en place des web services

En tant que responsable du service informatique du SEG, je souhaite que les applications mobiles accèdent aux données stockées sous SQL Server au travers de web services afin que les applications n’accèdent pas directement au SGBD pour une meilleure sécurité des données.

On souhaite obtenir à l’aide de ce web services :
• une liste d’objets « commune ».
• une liste des derniers relevés de chaque vanne (donc ceux de l’année précedente).

On souhaite également un web service qui enregistre les relevés dans la base de données sous SQL Server de l’application « gestconso ». La liste des relevés sera envoyée au web service au format json.





Mise en place de la base de données embarquée SQLite

En tant que responsable des applications mobiles, on souhaite que les données soient disponibles depuis le support mobile sans accès internet afin que les agents municipaux puissent saisir les relevés des compteurs vannes dans les secteurs où la connexion à internet est difficile.

Le modèle logique est dénormalisé pour optimiser l’application. Les contraintes de clés étrangères ne sont pas gérées par le SGBD ainsi que la clé primaire de la table « Releve » qui recevra sa valeur au moment de l’exportation dans la base de données distante.





Mettre en place le point d’entrée de l’application

Le point d’entrée de l’application se fera à partir d’un écran d’accueil, où on trouvera un menu qui est constitué de :

• L’item « Gérer les relevés des vannes » permettra de consulter les relevés et enregistrer de nouveaux relevés.
• L’item « Importer les données » permettra à l’agent d’importer soit toutes les données, soit le dernier relevé de chaque vanne depuis la base de données distante.
• L’item « Exporter les relevés » permettra à l’agent d’enregistrer tous les relevés de l’année en cours dans la base de données distante.





Gérer les données d’un relevé d’un compteur « vanne »

En tant que responsable du service des eaux, je souhaite que chaque agent puisse enregistrer les informations d’un relevé d’un compteur « vanne ».

Quand l’agent souhaite saisir un relevé, il choisit une commune, puis un secteur, puis la vanne. Quand l’agent clique sur une vanne, une liste des relevés est affichée (celui de l’année précédente et éventuellement celui de l’année en cours si le relevé a été réalisé). Sur cette liste, on affichera l’année et l’index des relevés.
L’utilisateur pourra saisir un nouveau relevé s’il n’existe pas déjà un relevé pour l’année en cours. L’agent souhaite que l’interface de saisie rappelle la date, l’index du dernier relevé ainsi que la référence et la marque de la vanne. La consommation sera également affichée après saisie du nouvel index qui doit être obligatoirement supérieur à l’index précédent.
